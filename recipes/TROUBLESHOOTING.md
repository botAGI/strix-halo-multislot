# Грабли, собранные лбом

Каждый пункт стоил реального времени отладки. Порядок — по частоте встречаемости.

## Docker / запуск

- **«unable to find group render»** — в образе llama.cpp нет именованных групп.
  `--group-add` только числовыми GID: `video=44`, `render=992` (Ubuntu; проверьте
  свои через `getent group video render`).
- **Vulkan не видит GPU в контейнере** — явно задайте ICD:
  `-e VK_DRIVER_FILES=/usr/share/vulkan/icd.d/radeon_icd.json`. Если на хосте
  когда-то стоял AMDVLK — удалите его ICD-файлы, у него cap 2 GiB на аллокацию,
  модели ≥30B не загрузятся.
- **Дефолт `--parallel` равен -1 (auto); в текущих исходниках server.cpp auto
  захардкожен в 4 слота + unified KV** (проверено на наших двух образах; до 11.2025
  дефолт был 1, потом принудительные 4, с 12.2025 официально -1/auto — резолюция
  auto может смениться в любой сборке). Сервер без флага уже
  мультислотный. Эффективный конфиг смотрите в `/props` → `total_slots`, а не в
  списке флагов. Мы на этом чуть не «открыли» несуществующий эффект тюнинга в 2.6×.

## MTP / спекулятивный декодинг

- **«[spec] failed to measure draft model memory»** — совпадающая ошибка описана
  в открытом НЕПОДТВЕРЖДЁННОМ issue llama.cpp #24350 (там другая платформа:
  CUDA/Windows); предложенный обход `-fit off` сработал и на Vulkan/RADV.
  `-ngld`/`--spec-draft-ngl` НЕ лечат.
- **`/props` показывает `"speculative.types":"none"` при работающем драфте** —
  верьте полям `draft_n`/`draft_n_accepted` в `timings` ответа.
- **Переименования флагов**: `--draft-max` → `--spec-draft-n-max`;
  `--ctx-size-draft` не существует.
- **`unknown model architecture: 'gemma4-assistant'`** — старые билды не знают
  арх MTP-драфтов Gemma. Нужен свежий образ.
- **MTP на батче — налог, не ускорение.** −31% уже на 4 потоках (при acceptance ~70%!).
  Физика: спекулятив выигрывает только пока decode memory-bound.

## Пределы GPU

- **Третий тяжёлый инстанс на GPU**: `vk::DeviceLostError` + dmesg
  «amdgpu: Not enough memory for command submission», exit 139. На дефолтном
  GTT-пуле (50% RAM) предел — два инстанса ~20GB. С `ttm.pages_limit` (94% RAM)
  пул вдвое больше, но мы предел не перепроверяли — осторожно.
- **~45-секундный одиночный столл** на батчёванном embed при `--parallel ≥ 8`:
  ровно один запрос за окно зависает на ~45с, p95 при этом чистый, пропускная
  не страдает. Пережил тюнинг ядра — это поведение llama.cpp/RADV, не хоста.
  dmesg чист. Если ваш клиент чувствителен к хвосту латентности — ставьте таймаут+ретрай.
- **Ложные GPU-reset под непрерывным батчингом** — `amdgpu.lockup_timeout=60000`
  (llama.cpp #21724).

## Бенчи

- **Кэш промптов подыгрывает.** Уникальная соль в каждом промпте + `cache_prompt: false`,
  иначе намеряете скорость кэша.
- **Агрегат считайте по серверным `timings`** (Σ predicted_n / makespan), не по
  клиентским секундомерам.
- **Пиновка ядра при A/B через ребут** — apt копит новые ядра; ребут без пина
  молча меняет вторую переменную (у нас в /boot ждало 7.0.0-28 при рабочем 6.17).
- **Соседние контейнеры на GPU** дают ~4% шума даже «в простое» — снимайте
  финальные числа на чистом узле.
